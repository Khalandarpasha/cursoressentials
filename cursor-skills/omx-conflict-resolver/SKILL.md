# Conflict Resolver (Phase 4)

## Purpose
Resolve merge collisions generated by parallel execution workers in a deterministic, auditable way.

## When to Use
- Multiple worker outputs touch overlapping files
- Cherry-pick from parallel branches fails
- Combined branch introduces semantic conflicts after merge

## Resolution Strategy

### Step 1: Pre-merge inventory
1. List candidate worker branches and commits.
2. Build overlap matrix by changed paths.
3. Prefer low-risk merge order:
   - non-overlap workers first
   - overlap workers later

### Step 2: Fast path (cherry-pick)
1. Cherry-pick non-overlapping commits in sequence.
2. Run lightweight checks after each pick.
3. If conflict-free, continue.

### Step 3: Overlap path (3-way merge)
1. Create temporary integration branch.
2. Perform 3-way merge for overlapping branches.
3. Resolve conflicts by rule:
   - preserve acceptance-criteria behavior first
   - preserve passing test behavior second
   - preserve style consistency third
4. Run focused tests/build/lint after each resolution block.

### Step 4: Escalation
Escalate to user when:
- conflict requires business/behavior decision
- two implementations are both valid but incompatible
- conflict resolution risks data loss or security regression

## Required Artifacts
- `.cursor/state/team/<teamId>/merge-report.json`
  - merge order
  - conflict files
  - resolution decisions
  - verification evidence

## Auto Mode Guidelines
- Continue automatically for mechanical conflicts.
- Pause and ask user for semantic/behavioral conflicts.
- Never force history rewrite operations unless explicitly requested.
- Keep conflict decisions documented in merge report.

## Verification Gates
- No unresolved conflict markers remain
- Affected tests pass
- Build/lint checks pass for changed modules
- Final diff matches accepted task outcomes
